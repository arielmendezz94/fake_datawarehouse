Chapter 01
The problem this project solves

Imagine you're the first data person hired at a fast-growing software company. The business runs on a dozen spreadsheets, a CRM, a billing system, and a payroll tool. The CEO wants to know: are we growing profitably? The VP of Sales wants to know which customers are about to churn. Finance needs the books closed by the 5th of every month. Nobody can answer any of it without hours of manual work.

This project builds the infrastructure that would fix that. CloudBridge Inc. is a fictional B2B software company — every customer, invoice, and salary number is synthetically generated — but the engineering is real. The goal was to build a complete, production-style analytics stack from a blank terminal: data generation, a database, transformation pipelines, and a business dashboard.

Think of it as a working prototype of the data engine behind any SaaS company's finance and operations reporting.

Chapter 02
First, we needed a company — so we built one in Python

Real company data is confidential. So the first engineering challenge was generating a fake company that behaves like a real one — with realistic growth curves, genuine accounting rules, and the kind of messy data that actually shows up in production.

A Python script using the Faker library invented 500 customers, 55 employees across four departments, 15 vendors (AWS, Stripe, Salesforce…), 8 product tiers, and four full fiscal years of transactions. But realistic data isn't just random numbers. Several deliberate design choices make it authentic:

What makes the data feel real
S-curve growth
Revenue grows slowly at first, then accelerates, mimicking how SaaS companies actually scale. New subscriptions follow a 2% monthly churn probability — not a fixed dropout date — so the MRR chart curves naturally instead of cliff-dropping.
Startup burn pattern
In 2021, costs exceed revenue — the classic venture-funded burn phase. By 2024 the net loss narrows to $933K as subscriptions scale. Engineering payroll compounds 3–5% every January, just like real salary bands.
Intentional data grime
Customer names and email addresses carry realistic noise — random casing, trailing spaces, occasional HTML entity typos. IDs and dollar amounts are always clean. This mirrors production databases and forces downstream cleaning logic to earn its keep.
Real accounting rules
Every invoice, payment, vendor bill, and payroll run produces balanced double-entry journal entries — for every debit there is an equal credit. The script asserts this integrity before writing a single row, so the general ledger is always mathematically correct.
By the numbers: 12 CSV files · ~210,000 total rows · ~160,000 general ledger entries · ~24,000 invoices · 720 vendor bills · ~2,700 payroll records · SEED = 42 (fully reproducible)
Chapter 03
The warehouse — a real database, running in one command

The generated CSVs are loaded into PostgreSQL 16, a battle-tested open-source database used by companies from startups to Fortune 500s. Rather than requiring a cloud account or paid service, the database runs locally inside Docker — a tool that packages software so it runs identically on any machine.

Anyone who clones this repo can have a running data warehouse in under a minute. The data persists across restarts, and the setup mirrors exactly how data teams run local development environments before deploying to the cloud.

Starting the database
One command
cd db && docker compose up -d
PostgreSQL 16 starts on localhost:5432. Connect with any SQL client — DBeaver, TablePlus, or psql.
Why Docker?
No manual installation, no version conflicts, no cloud bills. The named volume postgres_data ensures data survives container restarts — no re-seeding needed during development.
Chapter 04
Turning raw data into trusted answers — the dbt pipeline

Raw data is messy, inconsistent, and not shaped for answering business questions. dbt (data build tool) is the industry-standard way to transform it. You write SQL, dbt manages the execution order, tests the outputs, and documents everything. It's the difference between a pile of ingredients and a finished dish.

The project uses a three-layer medallion architecture — a design pattern borrowed from the best data teams at companies like Airbnb and Netflix. Each layer has a single job, and no layer ever reaches backwards to an earlier one:

A key design principle: the dashboard does no thinking. Every calculation — growth rates, year-to-date totals, gross margin — is computed in SQL inside dbt, where it can be tested and versioned. Chapter 05
The hard engineering problems — and how they were solved

Any competent engineer can join a few tables. The interesting work is in the edge cases — the problems that trip up real pipelines in production. Here are four that this project specifically solves.

Hard problem 01 — Missing months in time-series reports

If a product had no sales in March, a naive query skips March entirely. That breaks every month-over-month calculation downstream. The solution is a calendar spine: a list of every month cross-joined with every account, so zero-activity periods still appear as $0 rows.

rpt_pl_statement — calendar × account CROSS JOIN
-- Every month × every P&L account, guaranteed — even with zero activity
WITH calendar_spine AS (
  SELECT generate_series('2021-01-01', '2024-12-01', '1 month'::interval)::date AS entry_month
),
account_spine AS (
  SELECT account_name, account_type FROM silver.dim_accounts
  WHERE  account_type IN ('REVENUE', 'COGS', 'OPEX')
),
report_base AS (
  SELECT c.entry_month, a.account_name, a.account_type
  FROM  calendar_spine c
  CROSS JOIN account_spine a  -- guarantees no gaps in the time series
)
Hard problem 02 — Year-to-date totals that reset in January

YTD revenue should accumulate Jan–Dec, then reset to zero on January 1st. SQL window functions handle this elegantly — the PARTITION BY year clause forces the running total to restart each fiscal year automatically.

YTD running total (resets each year)
SUM(period_amount) OVER (
  PARTITION BY account_name,
               entry_year   -- resets Jan 1
  ORDER BY entry_month
) AS ytd_cumulative_amount
MoM and YoY variance with LAG
LAG(period_amount, 1) OVER (
  PARTITION BY account_name
  ORDER BY entry_month
) AS prior_month,

LAG(period_amount, 12) OVER (
  PARTITION BY account_name
  ORDER BY entry_month
) AS prior_year
Hard problem 03 — Making the P&L add up in any BI tool

Accounting has a sign convention problem: revenue is a credit (positive in accounting terms) while expenses are debits. If a BI tool naively sums all amounts, revenue and costs cancel each other out. The standard_balance_amount column pre-applies the correct sign so that a simple SUM() in Power BI always produces the right P&L — no CASE statements needed in the dashboard.

fct_journal_entries — sign convention baked in at the source
-- Revenue and liabilities: credits are positive (money coming in)
-- Assets and expenses: debits are positive (money going out)
CASE
  WHEN account_type IN ('REVENUE', 'LIABILITY', 'EQUITY')
       THEN credit_amount - debit_amount
  WHEN account_type IN ('ASSET', 'COGS', 'OPEX')
       THEN debit_amount  - credit_amount
END AS standard_balance_amount
-- Now SUM(standard_balance_amount) = correct signed P&L everywhere, no exceptions
Hard problem 04 — dbt config from the actual repo

The three-layer materialization strategy is declared once in dbt_project.yml and inherited by every model automatically. Bronze runs incrementally (fast, append-only); silver and gold rebuild fully (always fresh, always correct).

dbt_project/dbt_project.yml — actual file from the repo
# Each layer gets its own schema and materialization strategy
models:
  dbt_project:
    bronze:
      +materialized: incremental  # new rows appended; history untouched
      +schema:       bronze
    silver:
      +materialized: table        # full rebuild; business logic always fresh
      +schema:       silver
    gold:
      +materialized: table        # BI-ready; silver refs only
      +schema:       gold

Chapter 07
Run it yourself — the whole stack, on your machine, in minutes

The repo is fully self-contained. You need Docker, Python 3.11, and dbt-postgres installed. Everything else — the database, the data, the models, the dashboard — is in the repo.

01
Clone or download the repository
git clone https://github.com/arielmendezz94/fake_datawarehouse.git
Or use the Download ZIP button above — no Git required.
02
Start the PostgreSQL database in Docker
cd db && docker compose up -d
Requires Docker Desktop. Starts PostgreSQL 16 on localhost:5432. Takes about 10 seconds.
03
Generate the synthetic company data
cd ../faker && pip install faker pandas && python 02-generate.py
Prints a financial sanity-check summary, then writes ~210K rows across 12 CSVs to faker/data/
04
Load the CSVs into PostgreSQL
psql -h localhost -U admin -d dbt_db # then \copy each table from CSV
Any SQL client works — DBeaver and TablePlus both have GUI import tools if you prefer not to use the command line.
05
Run the dbt pipeline
cd ../dbt_project && dbt deps && dbt build
Runs all 23 models in dependency order and executes every data quality test. A green run means all 3 schemas are populated and verified.
06
Open the Power BI dashboard
open viz/dashboard.pbix
Requires Power BI Desktop (free). Connects directly to the gold schema — P&L, balance sheet, MRR, AR aging, and cash flow all pre-built.
